Plan de route du développement

Le guide pas à pas de Li·La, module par module : ce qu'il faut faire, dans quel ordre, avec quelles compétences, à quel coût, et comment vérifier que c'est bon. Pense-le comme ta feuille de chantier.

Comment l'utiliser

Chaque module a une fenêtre (quand le faire), une durée estimée (charge), les compétences mobilisées, les coûts d'exploitation, puis les étapes pas à pas, les points de vigilance, la validation et le livrable.

Les durées sont des hypothèses pour une développeuse seule. Elles servent à voir où le temps part, pas à t'enfermer. Le cockpit de pilotage reste la version vivante (cochage, avancement) ; ce document est la version « comment je m'y prends ».

Stack & principes transverses

Brique technique

  • Front : HTML / CSS / JS (PWA), déployé sur Netlify.
  • Back & données : Google Apps Script + Google Sheets (jusqu'à 100 adhérents), puis Supabase.
  • IA : API Anthropic (Claude), appelée uniquement via le proxy Apps Script.
  • Périphérie : Tally (formulaires/RSVP), MailerLite (emails), Stripe (paiement).
3 principes non négociables. 1) La clé IA reste côté serveur, jamais dans le navigateur. 2) La bibliothèque de prompts est ton IP : elle vit dans le proxy, invisible. 3) Li·La propose, l'agent valide : aucune écriture irréversible sans confirmation.

C1 · Juillet

Socle données

30 juin → 5 juil.≈ 3 jLaurence

Une base Google Sheets propre, prête à porter l'app, et documentée pour migrer vers Supabase sans tout refaire.

Compétences
  • Google Sheets (structure)
  • Apps Script (JS, lecture/écriture)
  • Notions de base de données
Coûts
  • Dev : ton temps (≈ 3 j)
  • Exploitation : gratuit (Google Workspace)

Étapes pas à pas

  1. Créer le classeur LILA_BASE avec un onglet par entité : Membres, Contacts, Notes, Contenus, Quotas, Logs.
  2. Poser les colonnes. Membres : id, email, offre, statut, date_inscription, quota_mois, conso_mois. Contacts : id, membre_id, nom, tel, email, type, date. Notes : id, membre_id, contact_id, texte, type, date, statut.
  3. Première colonne = id unique sur chaque ligne (via Utilities.getUuid() ou un compteur).
  4. Écrire les fonctions d'accès en Apps Script : getRows(sheet), appendRow(sheet,obj), updateRow(sheet,id,obj).
  5. Protéger toute écriture par LockService.getScriptLock() pour éviter les collisions.
  6. Documenter le schéma dans un onglet _SCHEMA : un onglet = une future table Supabase, une colonne = un futur champ.

Point de vigilance

Pense « tables » dès maintenant : pas de colonnes fourre-tout, des id partout, des liaisons par id (membre_id, contact_id). C'est ce qui rendra la bascule Supabase mécanique au lieu d'un chantier.

Validation

Écriture/lecture testée dans chaque onglet · deux écritures simultanées ne s'écrasent pas · le schéma est lisible par quelqu'un qui découvre la base.

Livrable

Classeur structuré + fonctions d'accès testées + schéma documenté.

C2 · Juillet

Proxy IA + quotas

6 → 12 juil.≈ 3 jLaurence

Le cœur technique du système : la clé IA reste serveur, chaque membre a un quota, le coût global est plafonné. Tout passe par ici.

Compétences
  • Apps Script avancé (doPost, UrlFetch)
  • API REST / JSON
  • Sécurité (gestion de clés)
  • Notions d'API LLM (tokens)
Coûts
  • Dev : ton temps (≈ 3 j)
  • Exploitation : API Anthropic à l'usage (au token) — le quota par membre la borne
  • Apps Script : gratuit

Étapes pas à pas

  1. Créer le point d'entrée doPost(e) : parser e.postData.contents en JSON → {type, email, payload}.
  2. Authentifier : retrouver l'email dans Membres, vérifier que le statut est actif et que l'offre couvre le type demandé.
  3. Vérifier le quota : conso_mois < quota_mois. Sinon, renvoyer un refus propre (message « quota atteint »).
  4. Router selon type (office_mail, note_struct, studio_post, studio_image) : c'est ici qu'on choisit le bon prompt dans ta bibliothèque, côté serveur.
  5. Appeler l'API via UrlFetchApp.fetch(), clé lue dans PropertiesService.getScriptProperties() — jamais en dur.
  6. Décompter les tokens consommés (champ usage de la réponse) dans Quotas + écrire une ligne Logs.
  7. Créer un trigger mensuel qui remet conso_mois à zéro.
  8. Gérer les erreurs : quota dépassé, IA indisponible, timeout → chaque cas renvoie un message clair.

Point de vigilance

Vérifie dans l'inspecteur réseau du navigateur qu'aucune trace de la clé n'apparaît. Le front n'appelle que ton URL /exec, jamais Anthropic directement.

Validation

Un appel renvoie une réponse · le quota se décrémente réellement · quota épuisé → blocage propre · clé invisible côté navigateur.

Livrable

Proxy en ligne, quota fonctionnel, clé protégée.

C3 · Juillet

Identification & Freemium

6 → 12 juil.≈ 2 jLaurence

Un non-abonné voit l'app, est tenté par les vitrines, mais n'accède au contenu qu'après identification et selon son offre. C'est le moteur du modèle.

Compétences
  • JS front
  • Logique d'accès / état
  • UX de « vitrine » convaincante
Coûts
  • Dev : ton temps (≈ 2 j)
  • Exploitation : gratuit

Étapes pas à pas

  1. Écran d'identification par email : l'email interroge Membres et renvoie l'offre du membre.
  2. Définir l'état d'accès : ava / office / studio / studioplus → ce qui est ouvert vs verrouillé.
  3. Construire les vitrines factices des pôles non souscrits : un aperçu réaliste et désirable, sans données ni action réelle.
  4. Verrouiller : au clic sur un pôle non souscrit → invitation à passer à l'offre supérieure (pas d'accès).
  5. Débloquer dynamiquement selon l'offre renvoyée (et plus tard, selon le paiement validé en C9).

Point de vigilance

La vitrine doit donner envie sans frustrer : montre la valeur, pas un mur. Et aucun vocabulaire interne à l'écran (pas de « quota », « offre studioplus » bruts).

Validation

Un non-abonné voit la vitrine sans accéder au contenu · le déblocage suit exactement l'offre du membre.

Livrable

Parcours verrouillé → identifié → débloqué selon l'offre.

C4 · Juillet

Lila Office — mail + CRM

13 → 19 juil.≈ 4 jLaurence

Le pôle qui justifie l'abonnement à 19€. Le module le plus lourd de la semaine — prévois-lui de la place.

Compétences
  • Gmail API / OAuth
  • JS, gestion d'état CRM
  • IA via proxy (rédaction)
Coûts
  • Dev : ton temps (≈ 4 j)
  • Exploitation : API à l'usage (rédaction seulement)

Étapes pas à pas

  1. Connexion à la boîte mail (Gmail API) : lecture des derniers messages, tri par priorité.
  2. Tri déterministe d'abord (labels, expéditeurs, mots-clés) — zéro coût IA. L'IA n'intervient qu'à la demande.
  3. CRM : afficher/éditer les Contacts, rattacher un mail à un contact, ajouter une note → écrite dans Notes.
  4. Assistant de rédaction : bouton « répondre » → appel proxy type=office_mail → brouillon proposé.
  5. Validation obligatoire avant envoi (Li·La propose, tu valides).

Point de vigilance

Garde le tri sans IA : c'est gratuit et instantané. Réserve l'IA à la rédaction et à l'analyse d'un mail ambigu, sinon les quotas fondent pour rien.

Validation

Mail trié + priorités OK · une note remonte dans le CRM · un brouillon cohérent est généré et validé avant envoi.

Livrable

Mail trié + CRM alimenté + brouillons générés.

C5 · Juillet

Lila Carnet — notes + mémo vocal

13 → 19 juil.≈ 3 jLaurence

Capturer une info en déplacement et la ranger au bon endroit, sans rien perdre.

Compétences
  • Web Speech API (+ repli texte)
  • JS front
  • IA via proxy (structuration)
Coûts
  • Dev : ton temps (≈ 3 j)
  • Exploitation : API à l'usage (structuration)

Étapes pas à pas

  1. Bouton de capture unique (vocal + texte), accessible partout dans l'app.
  2. Dictée via Web Speech API — avec un repli texte obligatoire (la dictée est capricieuse sur iOS).
  3. Envoi du texte brut au proxy type=note_struct → proposition structurée (contact / rappel / note).
  4. UI de rattachement : relier la note à un contact ou un dossier existant (par id).
  5. Validation avant enregistrement, puis écriture dans Notes.

Point de vigilance

Sur iPhone, la reconnaissance vocale peut être bloquée : le champ texte doit toujours fonctionner en repli, sinon le module est inutilisable en démo.

Validation

Dictée → proposition correcte · repli texte OK sur iOS · rien n'est enregistré sans validation.

Livrable

Dictée → proposition structurée → validée → rangée au bon endroit.

C7 · Juillet (déplaçable en V1.1)

Li·La Com'+ — image avancée

27 → 31 juil.≈ 2 jLaurence

Le haut de gamme visuel : traitement d'image et prompts archi/BTP. Poste le plus coûteux à l'usage — à surveiller.

Compétences
  • API de génération d'image
  • Prompting visuel archi/BTP
  • Réglage de quota serré
Coûts
  • Dev : ton temps (≈ 2 j)
  • Exploitation : API image (cher) — quota strict obligatoire

Étapes pas à pas

  1. Brancher l'API de traitement / génération d'image au proxy type=studio_image.
  2. Intégrer la bibliothèque de prompts archi/BTP (côté serveur, invisible).
  3. Poser un quota image serré distinct du quota texte.
  4. Afficher clairement la consommation restante au membre.

Point de vigilance

L'image coûte bien plus que le texte. Sans quota dédié strict, ce module peut manger toute la marge d'un abonnement à lui seul.

Validation

Un visuel produit est exploitable · le quota image borne réellement le coût.

Livrable

Visuels avancés + quota qui protège la marge.

PWA / Landing / Domaine / RGPD

27 → 31 juil.≈ 2 jLaurence

Rendre Li·La installable, visible et conforme. Inclut la landing simple et la prise de démo.

Compétences
  • PWA (manifest, service worker)
  • Déploiement Netlify, DNS
  • Bases RGPD
Coûts
  • Dev : ton temps (≈ 2 j)
  • Exploitation : domaine ≈ 12–15€/an + Netlify (gratuit au départ)

Étapes pas à pas

  1. Coque PWA : manifest.json + sw.js + bannière d'installation (icônes fond blanc pour iOS).
  2. Landing en une page : promesse + 3 bénéfices + 2 boutons (« Réserver une démo » / « Être prévenu »).
  3. Brancher le bouton démo sur Tally ou Calendly.
  4. Acheter / configurer le nom de domaine + DNS vers Netlify.
  5. Pages RGPD : mentions légales, CGV, politique de données (« vos données restent chez vous »).

Validation

App installable sur mobile · landing en ligne + prise de démo qui marche · documents RGPD publiés.

Livrable

App installable + landing en ligne + RGPD publié.
